¿Qué es PollaF?

PollaF es una aplicación web de predicciones del Mundial FIFA 2026, desarrollada en Django + PostgreSQL para la Universidad de América. Los participantes se registran con su correo @usa.edu.co, completan predicciones de los 104 partidos (72 de grupos + 32 eliminatorios) antes del inicio del torneo y compiten en un ranking global por puntaje acumulado.

48 equipos

12 grupos de 4 países. Datos fijos, pre-cargados con el comando poblar_mundial.

104 partidos

72 grupos + 16 R32 + 8 R16 + 4 QF + 2 SF + 1 TP + 1 FI. El usuario predice todos.

Bracket en cascada

Las fases eliminatorias se auto-construyen desde las predicciones de grupos del usuario.

Ranking global

Todos compiten en la misma tabla. Los puntos se actualizan en tiempo real al ingresar resultados.

Admin restringido

El administrador solo ingresa marcadores reales. Equipos, puntos y estructura son automáticos.

Bloqueo de torneo

Un botón en el admin congela todas las predicciones cuando el torneo comienza.

Patrón MVT: Django usa Model–View–Template. El Model define tablas ORM, la View contiene toda la lógica de negocio y construye el contexto, el Template renderiza HTML con ese contexto. El router (urls.py) hace el rol del Controller de MVC clásico.

MVC vs MVT — cómo se llaman las piezas en Django

La arquitectura más conocida en desarrollo web es MVC (Model – View – Controller). Django usa una variante llamada MVT (Model – View – Template), pero con los nombres cambiados de forma confusa. Esta tabla lo aclara:

Concepto Nombre en MVC Nombre en Django (MVT) Archivo en PollaF Qué hace
Los datos Model Model core/models.py Define las tablas de la base de datos. Igual en ambos patrones.
La lógica Controller View core/views.py Recibe la petición, consulta la BD, procesa datos, decide qué mostrar. Es el "cerebro".
Lo que ve el usuario View Template core/templates/*.html Recibe los datos procesados y los muestra como HTML en el navegador.
El enrutador (parte del Controller) Router core/urls.py Lee la URL de la petición y decide qué View llamar. Django lo separa en su propio archivo.
El truco: Django llama "View" a lo que MVC llama Controller, y llama "Template" a lo que MVC llama View. Los nombres están invertidos. Si vienes de otro framework esto es fuente de confusión eterna.

Ejemplo concreto: el usuario abre /ranking/

① Router — urls.py

Ve que la URL es /ranking/ y llama a la función ranking en views.py.

② View — views.py

Consulta la BD, ordena usuarios por puntos_totales, arma una lista con pos/nombre/puntos.

③ Template — ranking.html

Recibe la lista y la convierte en una tabla HTML con colores, posiciones y el usuario resaltado.

④ Respuesta

El HTML renderizado llega al navegador del usuario. La BD nunca fue tocada directamente desde el HTML.

Por qué se separó el "Controller" en dos piezas

En MVC tradicional, el Controller hace dos cosas: decide qué función ejecutar (routing) y ejecuta la lógica. Django las separó porque en una app grande con 50 URLs, tener todo el routing mezclado con la lógica se vuelve inmanejable. Al tenerlo en urls.py separado, puedes ver de un vistazo todas las rutas de la app sin leer código de lógica.

¿Por qué hay dos carpetas con nombre similar?

Es la parte más confusa para quien empieza. Cuando se crea un proyecto Django con el comando django-admin startproject pollaf, Django crea automáticamente una subcarpeta con el mismo nombre del proyecto para guardar la configuración. Resultado: la carpeta del escritorio se llama PollaF (con mayúscula, la creaste tú) y adentro hay una carpeta pollaf (minúsculas, la creó Django).

📁
PollaF/
La carpeta del escritorio. Contiene todo: el código, la configuración, la documentación. Es la raíz del proyecto.
La creaste tú.
⚙️
pollaf/
Configuración global: base de datos, apps instaladas, URL raíz, contraseñas. No tiene lógica de negocio de la app.
La creó Django automáticamente.
🏗️
core/
La app de Django. Aquí vive TODO: las tablas de la BD, las páginas, los formularios, los algoritmos de puntos y bracket.
La creaste tú con startapp core.
Analogía: Si PollaF fuera una tienda física → pollaf/ sería el acta de constitución (dirección, nombre legal, reglamento interno) y core/ sería el almacén completo con los empleados, los productos y las cajas registradoras. El acta dice que la tienda existe; el almacén es donde pasan las cosas.

Archivos en detalle — qué hace cada uno

📁 pollaf/ — configuración global

📁 core/ — la app completa

En Django, una "app" es un módulo autocontenido que agrupa modelos, vistas, URLs y templates de una funcionalidad. Este proyecto tiene una sola app: core.

ArchivoQué contiene / para qué sirve
models.py Las tablas de la base de datos. Cada clase Python aquí = una tabla en PostgreSQL. Define los 7 modelos: Pais (48 países), Partido (104 partidos), TorneoConfig (bloqueo), PerfilUsuario (participantes), Prediccion (predicciones), ClasificacionManualGrupo (clasificación de grupos personalizada por usuario), TerceroManual (orden manual de terceros del usuario).
views.py La lógica de cada página. Cada función aquí es una "vista": recibe la petición HTTP del usuario, consulta la BD, procesa datos, y entrega un contexto al template HTML. Son 9 vistas: home, eliminatorias, clasificacion_manual, mundial, ranking, llenar_prueba, registro, login, logout.
urls.py El mapa de URLs de la app. Conecta una dirección web con su vista correspondiente. Ej: /ranking/ → función ranking en views.py.
forms.py Los formularios con validación. Define RegistroForm (valida que el correo sea @usa.edu.co, que las contraseñas coincidan) y LoginForm (correo + contraseña).
admin.py La configuración del panel /admin/. Define qué puede y qué no puede hacer el administrador en cada tabla: qué campos ve, cuáles puede editar, qué acciones tiene disponibles.
scoring.py El algoritmo de puntos. La función calcular_puntos_partido() se ejecuta automáticamente cuando el admin ingresa un resultado. Calcula 0, 2 o 5 puntos para cada predicción y actualiza el total de cada participante.
bracket.py El bracket eliminatorio en cascada. Toma las predicciones de grupos de un usuario y calcula, paso a paso, quién jugaría en R32, R16, QF, SF, Final — todo derivado automáticamente.
terceros.py El asignador de mejores terceros. Decide, de los 12 terceros de grupo, cuáles 8 clasifican y a qué partido R32 va cada uno según las reglas FIFA.
terceros_tabla.py Las 495 combinaciones FIFA. Un diccionario Python gigante con todas las posibles combinaciones de grupos que pueden clasificar como tercer lugar, y la asignación correcta de slots. Evita recalcular en cada petición.
apps.py Metadata de la app (nombre, configuración interna). Django lo usa internamente; no se toca.
tests.py Archivo para pruebas automatizadas. Actualmente vacío; se puede usar para escribir tests unitarios.

📁 core/templates/ — los archivos HTML

Los templates son los archivos HTML que Django rellena con datos y envía al navegador. Usan una sintaxis especial con {% %} para lógica y {{ }} para mostrar datos.

¿Por qué hay dos niveles de carpetas? core/templates/core/ puede parecer redundante, pero es la convención de Django: la carpeta templates/ la detecta Django automáticamente, y la subcarpeta core/ evita conflictos de nombres si hubiera varias apps con un archivo base.html cada una.

📁 core/templatetags/ — funciones extra para los templates

Django no permite hacer operaciones complejas dentro del HTML (como acceder a un diccionario con una clave variable). Los "template tags" son funciones Python que se pueden llamar desde dentro del HTML con la sintaxis {{ variable|nombre_filtro }}. Se cargan con {% load core_extras %}.

Filtro en core_extras.pyPara qué sirve
get_itemAccede a un dict con una clave variable. Sin esto, Django solo permite claves fijas.
saved_in_groupCuenta cuántos partidos de un grupo ya tienen predicción (para el punto verde/amarillo del tab).
flag_urlConvierte código FIFA (ej: "COL") en URL de bandera PNG (flagcdn.com/24x18/co.png).
puntos_color / puntos_bg / puntos_iconoDevuelven colores CSS y texto según si el usuario sacó 5, 2 o 0 puntos en un partido.

📁 core/migrations/ — historial de cambios a la BD

Cada vez que modificas un modelo en models.py y corres python manage.py makemigrations, Django genera un archivo de migración aquí. Cada archivo registra exactamente qué cambió en la base de datos (qué tabla se creó, qué columna se agregó, etc.). Son como un historial de versiones de la BD.

No se editan a mano. Django los genera y los aplica automáticamente con python manage.py migrate.

📁 core/management/commands/ — comandos personalizados

Permite crear comandos propios que se ejecutan con python manage.py nombre_comando.

ComandoQué hace

Modelos Django — tablas en PostgreSQL

Todos los modelos viven en core/models.py. Django convierte cada clase en una tabla SQL via ORM. Los nombres de tabla siguen el patrón core_<nombreclase>.

1. Pais — core_pais

Tabla fija con los 48 países del Mundial 2026. Se carga una sola vez con el management command poblar_mundial y nunca es modificada por el usuario ni el admin.

class Pais(models.Model):

    grupo   = models.CharField(max_length=1)          # "A" … "L"

    class Meta:
        verbose_name_plural = "Paises"

3. TorneoConfig — core_torneoconfig

Singleton (pk=1 siempre). Controla si el torneo está abierto o bloqueado a predicciones. El método de clase get() usa get_or_create(pk=1) para garantizar existencia sin duplicados.

class TorneoConfig(models.Model):
    bloqueado    = models.BooleanField(default=False)
    bloqueado_en = models.DateTimeField(null=True, blank=True)

    @classmethod
    def get(cls):
        obj, _ = cls.objects.get_or_create(pk=1)
        return obj

    @classmethod
    def esta_bloqueado(cls):
        return cls.get().bloqueado

Partidos de grupos — distribución

Cada grupo tiene 6 partidos (todos contra todos). Los partidos 1–72 se distribuyen de forma que cada par de grupos comparte jornada. Los grupos A–L se alternan en las jornadas del calendario FIFA.

Regla de mejores terceros

De los 12 grupos, cada uno produce un equipo en tercer lugar. Solo 8 de esos 12 terceros avanzan a la Ronda de 32. Se seleccionan por orden de mérito:

  1. Puntos en fase de grupos
  2. Diferencia de goles (GF − GC)
  3. Goles a favor
  4. Victorias
  5. Fair play (tarjetas — implementado en el mundo real; en PollaF se usa hasta wins)

Asignación a slots de Ronda de 32

FIFA tiene una tabla de 495 combinaciones (C(12,8)) que indica, dado el conjunto exacto de 8 grupos cuyos terceros clasifican, a qué partido R32 va cada uno. Esta restricción existe porque los slots de tercer lugar en R32 solo aceptan equipos de ciertos grupos (para garantizar equilibrio geográfico).

Sistema de puntuación

5

ej: predijo 2-1, fue 2-1
2

Fase de grupos (GR)

  • El resultado puede ser: victoria local (L), empate (E), victoria visitante (V).
  • 5 pts si goles_local_pred == goles_local_real AND goles_visitante_pred == goles_visitante_real
  • 2 pts si el resultado (L/E/V) coincide pero los goles son distintos.
  • 0 pts si el resultado no coincide.

Fases eliminatorias (R32, R16, QF, SF, TP, FI)

  • No hay empate final: siempre hay un clasificado (si 90 min empatan → tiempo extra → penales).
  • Si el usuario predice empate en el marcador, debe elegir además quién clasifica (campo clasificado_pred).
  • El admin también registra clasificado_real cuando el marcador en 90 min es empate.
  • 5 pts: marcador exacto en 90 min Y clasificado correcto.
  • 2 pts: clasificado correcto pero marcador errado (o empate correcto pero clasificado incorrecto no aplica → 0).
  • 0 pts: clasificado incorrecto.
En eliminatorias, el "ganador real" (ganador_real) se determina así: si el marcador en 90 min es diferente → el equipo con más goles. Si empatan en 90 min → partido.clasificado_real (lo ingresa el admin). El usuario predice un marcador y si ese marcador es empate debe elegir quién avanza en clasificado_pred.

Registro y autenticación

Flujo de registro

  1. Usuario entra a /registro/ (GET) → ve el formulario.
  2. Envía POST con nombre, correo, contraseña x2.
  3. RegistroForm.clean_email() valida que termine en @usa.edu.co y que no exista ya.
  4. clean() valida que las dos contraseñas coincidan.
  5. Si válido: User.objects.create_user(username=email, email=email, password=...) + PerfilUsuario.objects.create(...)
  6. Redirige a /login/ con mensaje de éxito.

Flujo de login

  1. LoginForm extiende AuthenticationForm — usa Django's built-in auth.
  2. El username del formulario es el email (el campo se renombró).
  3. Django autentica con PBKDF2+SHA256 (hashing automático).
  4. Si user.is_staff → redirige a /admin/. Si es participante → redirige a /.

Sesiones

Django maneja la sesión con cookies firmadas (sessionid). El decorador @login_required garantiza que todas las vistas de participantes requieran autenticación; si no hay sesión activa, redirige a /login/.

Separación admin / participante

Los usuarios is_staff=True no tienen PerfilUsuario. El helper _get_perfil(request) captura la excepción RelatedObjectDoesNotExist y retorna None. Al inicio de cada vista protegida se verifica:

if request.user.is_staff:
    return redirect('/admin/')
perfil = _get_perfil(request)
if perfil is None:
    return redirect('core:login')

class EstrategiaEliminatoria:
    def calcular(self, gl_p, gv_p, gl_r, gv_r, ganador_pred=None, ganador_real=None):
        exact   = (gl_p == gl_r and gv_p == gv_r)
        correct = (ganador_pred is not None and ganador_real is not None
                   and ganador_pred.id == ganador_real.id)
        if exact and correct: return 5
        return 2 if correct else 0

def calcular_puntos_partido(partido):
    if not partido.jugado or partido.goles_local_real is None: return 0
    estrategia = _get_estrategia(partido.fase)
    gan_real   = _ganador_real(partido)
    preds      = list(Prediccion.objects.filter(partido=partido)
                      .select_related('usuario', 'clasificado_pred'))

                  for pred in preds]

    _recalcular_totales(usuario_ids)
    return len(preds)

Mejores terceros — cómo funciona todo el sistema

Esta es la parte más compleja del proyecto. Dos archivos trabajan juntos: core/terceros.py (lógica) y core/terceros_tabla.py (datos). Aquí se explica todo desde cero.

El problema que resuelve

En el Mundial 2026 hay 12 grupos. Cada grupo produce un equipo en tercer lugar. De esos 12 terceros, solo 8 clasifican a la Ronda de 32. Ahora bien: ¿en qué partido R32 juega cada tercero? No puede ser cualquier partido. FIFA tiene una restricción: cada slot de tercer lugar solo acepta equipos de ciertos grupos. Eso da 495 combinaciones posibles (C(12,8) = 495), y para cada una FIFA tiene una asignación exacta.

¿Por qué frozenset y no set normal?

Los diccionarios en Python requieren que sus claves sean hashables (inmutables). Un set normal es mutable — su contenido puede cambiar — por lo que Python no puede calcularle un hash estable y no puede usarse como clave. Un frozenset es exactamente igual que un set pero congelado: no puede cambiar, tiene hash fijo, y puede ser clave de diccionario.

# Esto falla — set es mutable, no hashable:
d = { {'A','B','C'}: 'valor' }
# TypeError: unhashable type: 'set'

# Esto funciona — frozenset es inmutable:
d = { frozenset('ABC'): 'valor' }  # OK

# Además, frozenset ignora el orden — estas tres claves son IGUALES:
frozenset('ABCDEFGH') == frozenset('HGFEDCBA') == frozenset(['A','B','C','D','E','F','G','H'])
# True — el orden no importa, solo los elementos

Esto es clave: no importa en qué orden lleguen los 8 grupos, el frozenset siempre produce la misma clave, y la búsqueda en el diccionario es O(1) — instantánea sin importar cuántas combinaciones tenga la tabla.

El solver de backtracking — cómo funciona paso a paso

Se usa como fallback si una combinación no está en la tabla (no debería pasar con las 495 precalculadas). El backtracking es un algoritmo de fuerza bruta inteligente: intenta asignar grupos a slots uno por uno y, si llega a un punto donde ningún grupo puede ir a un slot, retrocede (backtrack) al paso anterior y prueba otra opción.

def _resolver_con_backtracking(grupos: list) -> dict:
    slots   = [74, 77, 79, 80, 81, 82, 85, 87]  # orden fijo → resultado determinístico
    asignacion = {}
    usados     = set()                                   # grupos ya asignados

    def backtrack(idx):
        if idx == len(slots):
            return True                              # todos los slots asignados → solución
        slot     = slots[idx]
        elegibles = SLOTS_ELEGIBLES[slot]
        for grupo in grupos:                          # grupos en orden alfabético
            if grupo in elegibles and grupo not in usados:
                asignacion[slot] = grupo
                usados.add(grupo)
                if backtrack(idx + 1):              # intenta el siguiente slot
                    return True
                # no funcionó → deshacer y probar otro
                del asignacion[slot]
                usados.discard(grupo)
        return False                                 # ningún grupo sirve → backtrack

    backtrack(0)
    return asignacion

Principios SOLID aplicados

El código fue revisado y refactorizado para aplicar los principios SOLID donde son beneficiosos en el contexto de Django. Django usa el patrón Active Record (el modelo es el repositorio), lo que hace que algunos principios sean menos relevantes o directamente contraproducentes si se fuerzan.

PrincipioEstadoDónde aplica
S — Single Responsibility ✅ Aplicado Cada función hace una sola cosa: _ganador_real() solo determina el ganador, _guardar_puntos_predicciones() solo escribe en BD, _recalcular_totales() solo suma puntos. En admin, _fila_prediccion_html() y _tabla_predicciones_html() generan HTML sin tocar la BD. En views, _requiere_participante centraliza el guard, _guardar_predicciones_grupo() y _construir_standings_grupo() encapsulan lógica puntual.
O — Open/Closed ✅ Aplicado El módulo de scoring usa el patrón Strategy: EstrategiaGrupo y EstrategiaEliminatoria. Para agregar una nueva regla de puntuación (ej. un formato playoff diferente) se crea una nueva clase sin modificar el código existente. _get_estrategia(fase) actúa como factory.
L — Liskov Substitution ✅ Aplicado Ambas estrategias de scoring (EstrategiaGrupo, EstrategiaEliminatoria) tienen la misma firma de método calcular(gl_p, gv_p, gl_r, gv_r, **kwargs). Son intercambiables sin que el orquestador sepa cuál está usando.
I — Interface Segregation ⚠️ Parcialmente Python no tiene interfaces formales. El contrato implícito de calcular() en las estrategias cumple el espíritu del principio. No se fuerza una clase base abstracta porque sería burocracia sin beneficio real en este tamaño de proyecto.
D — Dependency Inversion ❌ No aplica Django usa Active Record por diseño: el modelo es su propio repositorio. Añadir una capa de repositorios abstracta rompería la ergonomía del ORM sin ningún beneficio práctico. Es una decisión consciente de no aplicarlo, no un olvido.

CSRF

Todos los formularios POST incluyen {% csrf_token %}. Django verifica el token automáticamente; si falta o no coincide, retorna 403. Las vistas personalizadas del admin usan admin_site.admin_view() que incluye verificación CSRF.

Sesiones

Almacenadas en la BD (django_session table). El ID de sesión va en cookie sessionid (HTTP-only). Expiración configurable con SESSION_COOKIE_AGE (default 2 semanas).
